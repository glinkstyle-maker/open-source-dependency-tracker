### 2.4 复杂查询模块

#### 按组件名查询所有关联漏洞
```sql
SELECT c.name, v.cve_id, v.severity, cv.status
FROM components c
INNER JOIN component_vulnerabilities cv ON c.id = cv.component_id
INNER JOIN vulnerabilities v ON cv.vulnerability_id = v.id
WHERE c.name = ?
```
✅ 快速定位特定组件的所有已知漏洞

#### 按项目名查询所有依赖及漏洞
```sql
SELECT p.name, c.name, v.cve_id, v.severity, cv.status
FROM projects p
INNER JOIN project_components pc ON p.id = pc.project_id
INNER JOIN components c ON pc.component_id = c.id
LEFT JOIN component_vulnerabilities cv ON c.id = cv.component_id
LEFT JOIN vulnerabilities v ON cv.vulnerability_id = v.id
WHERE p.name = ?
```
✅ 全面展示项目的依赖链和安全风险

#### 查找所有高危未修复漏洞
```sql
SELECT c.name, v.cve_id, v.severity
FROM components c
INNER JOIN component_vulnerabilities cv ON c.id = cv.component_id
INNER JOIN vulnerabilities v ON cv.vulnerability_id = v.id
WHERE v.severity = 'CRITICAL' AND cv.status = '未修复'
```
✅ 用于安全审计，快速定位关键风险

#### 统计每个项目的漏洞数（GROUP BY）
```sql
SELECT p.name, COUNT(DISTINCT v.id) as total,
       SUM(CASE WHEN v.severity = 'CRITICAL' THEN 1 ELSE 0 END) as critical_count
FROM projects p
INNER JOIN project_components pc ON p.id = pc.project_id
INNER JOIN components c ON pc.component_id = c.id
LEFT JOIN component_vulnerabilities cv ON c.id = cv.component_id
LEFT JOIN vulnerabilities v ON cv.vulnerability_id = v.id
GROUP BY p.id, p.name
```
✅ 项目风险评分和对比分析

### 2.5 数据完整性保证

#### 外键约束
```
ProjectComponents:
  - project_id → projects(id) [ON DELETE CASCADE]
  - component_id → components(id) [ON DELETE CASCADE]

ComponentVulnerabilities:
  - component_id → components(id) [ON DELETE CASCADE]
  - vulnerability_id → vulnerabilities(id) [ON DELETE CASCADE]
```
✅ 防止数据孤立，确保引用完整性

#### 事务管理
```java
// 确保添加组件和关联漏洞的原子性

    componentVulnerabilityDAO.add(...);
    tm.commit();

    tm.rollback();
    throw e;
}
```
✅ ACID特性保证数据一致性

## 四、主要挑战与解决方案

### 4.1 多对多关系处理

**挑战：** 一对多的JOIN结果中，主表数据重复

```
项目A - 组件1 - 漏洞1
项目A - 组件1 - 漏洞2  ← 项目A重复
```

**解决：** 
- 使用String[]数组接收原始数据
- 在UI层进行数据分组和去重
- 使用DISTINCT关键字进行数据库级去重

### 4.2 NULL值处理

**挑战：** LEFT JOIN可能返回NULL值

```
项目 → 组件(无漏洞) → LEFT JOIN结果中v.cve_id为NULL
```

**解决：**
```java
if (row[3] == null) {
    // 该组件暂无已知漏洞
} else {
    // 显示漏洞信息
}
```

### 4.3 级联删除的平衡

**挑战：** 删除项目时是否删除组件

**解决：** 
- 删除项目 → 删除ProjectComponents（级联）
- 保留Components表记录（其他项目可能依赖）
- 实现了"软删除"的效果

### 4.4 版本管理复杂性

**挑战：** 项目中使用的组件版本与最新版本不同

```
项目使用Log4j 2.14.0（有漏洞）
最新版本Log4j 2.17.1（已修复）
```

**解决：**
- component_version字段记录项目实际使用版本
- affected_version_start/end定义漏洞影响范围
- 精确匹配项目风险
